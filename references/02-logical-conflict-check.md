# 02 — 合流前逻辑冲突检查（Phase 2，本 skill 核心）

git 的冲突检测**只看「同一文件的相邻/重叠行是否被双方都改了」**。它完全无法发现这类情况：主干在 A 文件改了一个函数的签名，分支在 B 文件按旧签名新增了一个调用方——merge 会"干净"通过，但代码已经逻辑损坏。

本阶段的目标：在 merge 之前，**主动找出这些 git 不会报告的语义冲突**，登记成预案，供 Phase 4/5 落地修正。

---

## §2.0 文件分类：单边修改快速放过 vs 双边修改深度检查

`scripts/collect_diffs.sh` 输出已将改动文件分为三类，不同类别采用不同检查策略：

### A 类：仅分支侧修改的文件（直接放过）
- 这是分支自己的新增/修改，主干从未触碰，不存在合流冲突可能。
- **动作**：直接放过，不纳入逻辑冲突检查。在 `LOGICAL_CONFLICTS.md` 的"检查范围"中注明跳过即可。

### B 类：仅主干侧修改的文件（快速放过）
- 主干改了，但分支侧对这些文件没有本地修改。
- **快速放过规则**：只需检查**分支侧是否存在对这些文件的跨文件依赖**（即分支侧代码 import / 调用 / 消费了主干修改文件中的符号、接口、契约），若存在依赖关系则按 C 类处理。
- **无跨文件依赖时**：直接放过，标记为"快速放过，无跨文件依赖"。
- **有跨文件依赖时**：升级为 C 类，对涉及的依赖点按 §2.2 的 7 类清单排查。

### C 类：双方共同触碰的文件（深度检查，保持现有流程）
- 主干和分支都修改了该文件，是行冲突和语义冲突的高发区。
- **动作**：逐文件精读 diff，按 §2.2 的 7 类语义冲突清单逐类排查，**无论 git 是否报告行冲突**。
- 即使 git 无行冲突（干净合入场景），两侧修改仍可能产生逻辑覆盖或语义矛盾，必须深度检查。

```
判断流程图：

改动文件
  ├─ 只有分支修改？ → A 类 → 直接放过
  ├─ 只有主干修改？ → B 类 → 有跨文件依赖？ → 是 → 升级为 C 类
  │                         └─ 否 → 快速放过
  └─ 双方都修改？   → C 类 → 深度检查（7 类清单 + 无论是否有行冲突）
```

---

## §2.1 收集双方改动

用辅助脚本一把收集（token 高效）：

```bash
bash scripts/collect_diffs.sh origin/<主干>      # merge-base 自动计算；也可显式传第二参
```

脚本输出主干侧 diff、分支侧 diff、双方共同触碰文件、主干侧 rename/delete 四部分概览。也可手工执行下列命令：

```bash
# 主干侧改动（merge-base → 主干）：线上/主干这段时间改了什么
git diff --stat <merge-base> origin/<主干>
git diff <merge-base> origin/<主干> -- <感兴趣文件>

# 分支侧改动（merge-base → 分支 HEAD）：本分支这段时间改了什么
git diff --stat <merge-base> HEAD
git diff <merge-base> HEAD -- <感兴趣文件>

# 双方共同触碰的文件（高风险交集，优先精读）
comm -12 <(git diff --name-only <merge-base> origin/<主干> | sort) \
         <(git diff --name-only <merge-base> HEAD | sort)
```

> 重点关注三类对象：
> 1. **C 类：双方都改过的文件**（行冲突高发，且语义冲突高发）——逐文件精读 diff。
> 2. **B 类中"被主干改了接口/契约、被分支消费"的跨文件关系**（行冲突不发生，但语义最危险）——升级为 C 类处理。
> 3. **A 类：仅分支侧修改**与 **B 类中无跨文件依赖的文件**——快速放过，不深度检查。

## §2.2 七类语义冲突清单（逐类排查）

对主干侧改动逐项判断"分支侧是否依赖了它的旧形态"，反之亦然。每命中一项登记到 `LOGICAL_CONFLICTS.md`。

### 类 1：函数/方法签名变更
- 主干改了函数名、参数顺序/类型/数量、返回值、是否报错；分支新增或保留了按**旧签名**的调用。
- 排查：对主干改动中变更签名的符号，在分支侧 `grep` 其调用点。

### 类 2：数据结构/字段变更
- 主干删除/重命名/改类型了 struct 字段、DB 列、proto 字段、JSON key、枚举值；分支仍读写**旧字段**。
- 排查：主干删除/改名的字段名，在分支侧全局搜索是否仍被引用。

### 类 3：接口/契约语义变更
- 主干改了 HTTP/RPC 路由、请求/响应结构、错误码含义、状态机取值、MQ 消息格式；分支按**旧契约**实现了生产者/消费者。
- 排查：契约定义文件（router、proto、错误码表、消息定义）双方是否各自演进。

### 类 4：行为/默认值/常量语义变更
- 主干改了某个常量、默认值、开关、阈值、配置项的**含义或默认行为**（行没冲突，但语义反了）；分支的逻辑建立在旧含义上。
- 典型：主干把 `enabled` 默认从 false 改 true、把超时单位从秒改毫秒、反转了某布尔判断。

### 类 5：模块重构 / 文件搬迁 / 删除
- 主干把某模块拆分/合并/移动/删除/改包路径；分支基于**旧位置**新增了代码或 import。
- 排查：主干 diff 中的 `rename`/`delete`/包路径变更；分支侧对应 import 是否还指旧路径。

### 类 6：共享资源 / 全局状态 / 副作用
- 双方都改了同一全局变量、单例、初始化顺序、中间件链、DI 注册、全局配置加载；行不冲突但运行时互相覆盖。
- 典型：双方都往同一个注册表注册、都改了 `init()`、都改了同一中间件顺序。

### 类 7：依赖 / 构建 / 迁移脚本
- 主干升级/降级了依赖版本、改了构建配置、加了 DB migration；分支引入了与新版本**不兼容**的用法，或两边的 migration 版本号/顺序冲突。
- 排查：`go.mod`/`package.json`/`pom.xml` 双方改动；`migrations/` 目录双方新增项。

## §2.3 风险分级

每条登记项标注风险等级：

| 等级 | 含义 | 处理 |
|---|---|---|
| 🔴 高 | 必然破坏：旧形态已被删除/改变，分支强依赖 | 必须在 Phase 4/5 修正；STOP 时重点向用户说明 |
| 🟡 中 | 可能破坏：语义变化，需人工确认行为是否仍正确 | Phase 5 逐条核对，必要时跑测试 |
| 🟢 低 | 疑似相关但大概率无碍 | 记录备查，Phase 5 抽查 |

---

## §2.4 产出与 STOP

- 按 `templates/LOGICAL_CONFLICTS.md` 写入 `.qiqskills/<仓库名>-<分支名>/LOGICAL_CONFLICTS.md`，每条含：编号、冲突类别、主干侧改动、分支侧改动/依赖、风险等级、合流后修正预案。
- **快速放过记录**：即使"未发现逻辑冲突"，也要在文档中写明：
  - A 类文件（仅分支修改）清单及其放过理由。
  - B 类文件（仅主干修改）清单，标注是否检查了跨文件依赖及结论。
  - C 类文件已按 7 类逐类排查，并列出已检查的 diff 作为证据。
- **STOP & CONFIRM**：把清单贴给用户。🔴 高风险项必须与用户达成处理共识后，才进入 Phase 3 merge。

---

## 反作弊提示

- 不允许只跑 `git diff --stat` 看个文件数就宣称"无逻辑冲突"。7 类清单必须对 C 类文件逐类有结论。
- C 类"双方共同改动的文件"必须逐个精读 diff，不能只看文件名。
- B 类"仅主干修改的文件"必须检查跨文件依赖（分支侧是否消费了主干修改的符号/接口），不能直接跳过。
- A 类"仅分支修改的文件"可以直接放过，无需深度检查。
- 跨文件的"接口被改 + 调用方未改"是最容易漏的，务必对主干变更的符号在分支侧反查引用。
