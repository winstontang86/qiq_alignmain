# 04 — 合流后整体审查（Phase 5）

merge 提交成功 **不等于** 对齐成功。本阶段做合流后的整体回归审查，重点回答两个问题：**是否覆盖了线上（主干已有）功能？是否引入了 bug？** 并把 Phase 2 登记的逻辑冲突逐条核对消化。

---

## §5.1 回归 Phase 2 逻辑冲突预案（最重要）

逐条打开 `LOGICAL_CONFLICTS.md`，对每一条：

- 🔴/🟡 项：在合流后的代码里**实际定位**对应位置，确认旧形态的依赖已按主干新形态修正。
  - 签名变更类：`grep` 被改符号的所有调用点，逐个确认参数/返回已适配。
  - 字段变更类：搜索被删/改名字段，确认无残留旧引用。
  - 契约/语义类：确认生产者/消费者、判断逻辑与主干新语义一致。
  - 重构搬迁类：确认 import/包路径指向主干新位置。
  - 共享状态类：确认注册/初始化/中间件顺序合并后无相互覆盖。
  - 依赖/迁移类：确认依赖版本统一、migration 顺序无冲突。
- 在 `POST_MERGE_REVIEW.md` 标注每条的最终状态：✅ 已消化 / ⚠️ 待人工 / ❌ 仍破坏。
- 注意：这些点 **git 不会报错**，必须靠搜索+阅读+测试主动核对。

---

## §5.2 覆盖线上功能检查

防止合流"把主干（线上在跑）的功能/修复又改回了旧样子"：

- 重点看 **干净合入**与 **§4 中采用了"偏向分支侧"解决**的位置。
- 对照主干侧 diff（`git diff <merge-base> origin/<主干>`），确认主干新增/修复的关键逻辑在合流结果中**仍然存在且生效**。
- 抽查方式：
  ```bash
  # 合流结果相对主干，是否意外"删掉/改回"了主干的内容
  git diff origin/<主干>...HEAD          # 三点：展示分支相对 merge-base 的净变化
  ```
  逐项确认这些差异都是"分支本应有的改动"，没有混入"把主干功能改回去"的内容。

---

## §5.3 引入 bug 检查（客观验证优先）

- **构建/编译**：跑项目构建（如 `go build ./...` / `npm run build` / `mvn compile` 等，按仓库实际命令）。
- **静态检查**：跑仓库已配置的 lint / vet / 类型检查。
- **测试**：跑受影响范围的测试或全量测试；优先覆盖 Phase 2 标注的高风险模块。
- **接口一致性抽样**：对签名/字段/契约变更点，grep 调用方与定义方是否一致。
- 把每项命令与结果（通过/失败/跳过原因）登记到 `POST_MERGE_REVIEW.md`，让"无 bug"成为客观结论而非主观判断。
- 构建/测试失败 → 视为合流未完成，定位修复（仍遵守不借机重构、存疑交人工的原则）后重跑。

---

## §5.4 处理被暂存的改动

- 若 Phase 1 做了 stash：
  ```bash
  git stash list                          # 找到本次对齐的 stash 引用
  git stash pop stash@{N}                 # 恢复（可能再次产生冲突）
  ```
  - pop 后若产生冲突，按 `references/03-merge-and-resolve.md` §4 的同样两条硬约束处理并记录。
  - 提示用户：是否现在恢复，或保留 stash 稍后处理。
- 若 Phase 1 做了 WIP commit：提示可用 `git reset --soft HEAD~1` 将 WIP 改动还原为工作区改动。

---

## §5.5 产出与 STOP

按 `templates/POST_MERGE_REVIEW.md` 写入 `.qiqskills/<分支名>/POST_MERGE_REVIEW.md`，包含：

- 逻辑冲突预案逐条消化结果。
- 覆盖线上功能检查结论。
- 引入 bug 检查：构建/lint/测试命令与结果清单。
- stash/WIP 恢复处理情况。
- 遗留风险与建议。

**STOP & CONFIRM**：把审查结论交用户确认。存在 ❌ 仍破坏 / ⚠️ 待人工 / 构建测试未过的项时，**不得宣告"对齐完成"**。
